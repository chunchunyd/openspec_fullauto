# Git 提交信息规范

本文档约束本项目在主项目代码仓库与 `openspec/` 规格仓库中的提交信息写法，目标是同时兼顾以下几点：

- 在 `git log --oneline` 中一眼看清这次提交对应哪个 `change-id`
- 在需要排查问题、理解接手背景或回溯实现取舍时，能快速读到本次提交的改动范围与实现思路
- 让多 agent、断点续跑和系列 step 的交接过程保持稳定，不依赖模糊的 `WIP` 或口头上下文

## 1. 适用范围

- 适用于所有与 OpenSpec change 直接对应的提交
- 适用于主项目代码仓库与 `openspec/` 规格仓库
- 适用于人工提交、agent 提交和自动化整理继承现场后的提交

## 2. 总体原则

- 每个提交的标题都必须以精确的 `change-id` 开头
- 提交标题保持简短，方便扫描历史
- 提交正文默认必须写清“本次改了什么”和“为什么这样改”
- 除 `change-id`、分支名、路径、命令、`What`、`Why`、`Notes` 这类关键标识外，提交信息默认使用中文
- 提交信息不得附带 `Co-authored-by`、`Generated-by` 或等价协作署名尾注
- 不使用 `WIP`、`misc`、`update`、`fix stuff` 这类无法帮助排查问题的模糊标题

## 3. 标题格式

标题固定写法：

```text
<change-id>: <本次提交的明确结果>
```

要求：

- `change-id` 必须完整，不要缩写
- 冒号后直接写本次提交的结果，不要再补第二个前缀
- 标题应描述“已经完成了什么”，而不是“准备做什么”
- 标题默认控制在一行内，避免写成过长段落

推荐示例：

```text
public-content-step-02: 补齐帖子详情读取模型
users-part1-step-02: 收紧自我资料更新字段边界
admin-audit-part1-step-04: 接通用户封禁动作的审计留痕
```

反例：

```text
WIP
update code
fix bug
public-content-step-02: 一些修改
```

## 4. 正文格式

默认使用如下结构：

```text
<change-id>: <本次提交的明确结果>

What:
- ...
- ...

Why:
- ...
- ...

Notes:
- ...
```

要求：

- `What` 必填，用中文写清这次实际改动的关键结果
- `Why` 必填，用中文写清关键取舍、边界或实现原因
- `Notes` 选填，用于补充接手提示、剩余限制或后续跟进点
- 正文描述结果与原因，不写机械性的文件清单
- 如果本次提交很小，`What` 与 `Why` 各 1 条即可；不要为了凑格式写空话
- 禁止添加Co-Author信息

推荐示例：

```text
public-content-step-02: 补齐帖子详情读取模型

What:
- 增加公开帖子详情查询入口与 DTO 映射
- 覆盖帖子不存在和未发布场景的受控返回

Why:
- 让详情读取边界与现有公开列表读取保持一致
- 避免在 service 外重复拼装可见性判断

Notes:
- 当前计数仍复用现有投影，后续可在 interactions 读模型稳定后再下沉
```

## 5. 常见场景写法

### 5.1 普通实现提交

用于完成一个或少数几个紧密相关任务后的阶段性提交。

```text
<change-id>: <补齐的能力或边界>
```

正文按 `What` / `Why` / `Notes` 模板填写。

### 5.2 继承现场整理提交

用于接手已有 worktree 时，把继承下来的 staged / unstaged / untracked 改动整理成可继续推进的基线。

推荐标题：

```text
<change-id>: 整理继承的实现现场
```

正文应明确说明：

- 保留并提交了哪些已有改动
- 清理了哪些确认无效的残留
- 为什么这样整理后可以继续推进

### 5.3 交接前收口提交

用于实现分支在切回或合并回 `series/<series-prefix>` 前的最后一次收口。

推荐标题：

```text
<change-id>: 完成交接前收口
```

正文应明确说明：

- 本次收口补齐了哪些剩余实现或验证结果
- 为什么现在已经适合交给上层集成线继续处理
- 如仍有已知限制，用 `Notes` 写清

### 5.4 文档或规则提交

如果本次提交主要修改 README、流程说明、脚本入口说明或 OpenSpec 协作文档，标题应直接写文档结果，不要假装成代码实现提交。

推荐示例：

```text
<change-id>: 更新提交规范文档
<change-id>: 补充自动化流程的限流重试说明
```

## 6. 与系列分支的关系

- 系列 step 的实现分支应保持小步提交
- 在切回或合并回 `series/<series-prefix>` 前，实现分支应已经完成必要提交，并保持工作区干净
- `series/<series-prefix>` 上的 squash merge 结果提交继续使用简洁标题，通常直接使用 `change-id`
- 详细实现思路主要保留在实现分支上的阶段性提交正文里，供开发期调试、排查和接手使用

## 7. 与 OpenSpec 的关系

- 提交正文帮助解释“本次具体怎么做、为什么这样做”
- `proposal.md`、`design.md`、`tasks.md` 与 spec delta 仍然是长期真相源
- 如果某次提交中的取舍已经上升为稳定边界或显式约束，应同步回写到对应 OpenSpec 文档，而不是只留在提交信息里

## 8. 在 tasks.md 中如何引用

凡是 `tasks.md` 中出现提交类任务，任务文本都应显式引用本文档，例如：

```md
- [ ] 按 `openspec/docs/git-commit-guidelines.md` 中的约定提交本阶段改动，并确保提交标题与正文完整说明本次结果和原因
```

或：

```md
- [ ] 按 `openspec/docs/git-commit-guidelines.md` 中的约定整理继承现场并提交，确认无效残留已安全清理
```

要求：

- 不能只写“及时 commit”
- 不能只写“提交代码”
- 必须明确要求参考本文档

## 9. 当前默认结论

- 标题短，正文清楚
- 标题必须带 `change-id`
- 正文默认使用 `What` / `Why` / `Notes`
- 除关键标识外默认使用中文
- 提交类任务必须显式引用本文档
