# 设计说明：运行时环境加载与数据库配置校验修正

## 目标

本次设计的目标非常聚焦：

1. 让 API 与 worker 在本地开发时稳定加载根目录 `.env`
2. 让数据库连接配置错误在启动阶段暴露，而不是拖到 readiness 或业务请求阶段
3. 把当前模糊的底层驱动报错，收敛成明确的运行时配置错误

## 当前问题

### 1. 运行时没有可靠加载 `.env`

当前 `apps/api/src/main.ts` 与 `apps/worker/src/main.ts` 只负责创建 Nest 应用并监听端口，没有明确的本地环境加载步骤。

同时，根 README 当前写法会让开发者自然地理解为：

1. 复制 `.env`
2. 执行 `pnpm dev-api` / `pnpm dev-worker`
3. 服务即可正常读取数据库与 Redis 配置

但代码层面并没有把这条假设真正变成运行时保证。

### 2. `libs/database` 直接消费未校验的 `DATABASE_URL`

当前 `PrismaService` 在构造 `pg` Pool 时直接读取：

- `process.env.DATABASE_URL`

然后立刻把该值交给 `pg` adapter。

这意味着：

- 如果环境变量未加载，`connectionString` 可能为 `undefined`
- 如果值包含异常空白或包裹字符，也不会先被规范化
- 错误会以底层驱动行为的形式在首次连接时暴露

### 3. `/health/ready` 只是把已有问题更早暴露出来

health change 修正后，API 与 worker 的 readiness 都会真实执行：

- 数据库 `SELECT 1`
- Redis `PING`

因此 readiness 现在会触发真正的数据库连接流程。

这不是 readiness 本身的缺陷，而是它首次把“数据库配置不可靠”这件事稳定暴露了出来。换句话说，health 并没有把数据库弄坏，而是把先前被掩盖的配置问题提早暴露了。

## 根因判断

本次问题最合理的根因应视为：

- **运行时配置加载缺失或不稳定**
- **数据库连接字符串缺少显式校验与清晰错误**

而不是将其归因于健康检查逻辑本身。

`pg` 的 `SASL: SCRAM-SERVER-FIRST-MESSAGE: client password must be a string` 这类报错，本质上说明底层拿到的认证参数不满足预期；在当前代码结构下，这通常意味着连接配置并没有在应用启动前被可靠准备好。

## 设计原则

### 1. 本地 `.env` 加载必须成为显式运行时行为

本地开发模式下，服务端运行时必须明确知道从哪里加载环境变量，而不是依赖贡献者手工 export、IDE 附带配置或隐式 shell 状态。

本次 change 应明确：

- monorepo 根 `.env` 是本地开发默认配置来源
- `apps/api` 与 `apps/worker` 都必须遵循同一套发现规则
- 部署环境注入的环境变量优先级高于本地 `.env`

### 2. 数据库配置必须先校验，再接线

`libs/database` 不应把原始 `process.env.DATABASE_URL` 直接无条件交给底层驱动。

至少应在构造连接前完成：

- 非空校验
- 去除意外空白
- 去除包裹型引号
- 最小格式校验（例如必须是 `postgresql://` 或 `postgres://`）

如果校验失败，应该抛出仓库自定义的清晰配置错误，而不是等待请求链路命中底层驱动异常。

### 3. 配置错误应在启动阶段失败快速

数据库连接配置属于关键启动依赖。

如果它无效，应用更合理的行为是：

- 启动失败
- 输出明确错误
- 提示开发者检查 `.env`、启动脚本或环境注入方式

而不是让应用先启动成功，再在 `/health/ready` 或业务请求中随机暴露异常。

## 建议实现

### 方案概览

本次 change 建议以最小方式完成：

1. 为 API 与 worker 增加统一的运行时环境加载入口
2. 在 `libs/database` 中提取数据库连接字符串解析/规范化 helper
3. 由该 helper 对 `DATABASE_URL` 做显式校验并返回规范化后的字符串
4. `PrismaService` 只接收已经通过校验的连接字符串

### 运行时环境加载

实现方式可以是：

- 使用 NestJS 标准配置模块在应用启动阶段加载 `.env`
- 或在应用最早入口建立统一的本地环境加载 helper

无论采用哪种具体方式，本次 change 都要求满足以下结果：

- `pnpm dev-api` 与 `pnpm dev-worker` 在本地开发下都能发现根 `.env`
- 不要求贡献者额外手工 export `DATABASE_URL`
- 行为在 API 与 worker 之间保持一致

### `DATABASE_URL` 规范化与校验

建议在 `libs/database` 增加一个小型 helper，例如：

- `resolveDatabaseUrl()`
- 或 `getDatabaseUrlOrThrow()`

该 helper 的职责应包括：

- 从运行时读取原始 `DATABASE_URL`
- 对字符串做 `trim`
- 如果值被单引号或双引号完整包裹，则去除外层包裹
- 对空值、空字符串或明显无效的协议前缀抛出清晰错误

这一步的目标不是完整重写 URL parser，而是把最常见、最误导人的配置错误挡在驱动层之前。

### 健康检查边界

本次 change 不重新定义 readiness 契约。

health 仍然应该继续做真实依赖检查，但它不应该成为“第一次发现启动配置就错了”的地方。配置错误应在进程初始化时就暴露并终止启动。

## 测试与文档

本次 change 至少应覆盖：

- 数据库连接字符串规范化与异常输入的单元测试
- 启动阶段在缺失或无效 `DATABASE_URL` 下的失败行为验证
- README 中本地开发与环境变量加载方式说明
- 必要注释，解释为何需要在数据库层做 fail-fast 配置校验

## 风险与取舍

### 风险

- 一旦启用 fail-fast，先前“勉强能启动但迟早会在请求时出错”的场景会更早失败
- 某些依赖特殊启动方式的本地脚本可能需要同步调整

### 取舍

这是有意的取舍。

对当前阶段来说，更早、更明确地失败，比把问题推迟到 readiness 或业务访问时再暴露要可靠得多，也更符合后续基建持续收敛的方向。
