# 变更提案：运行时环境加载与数据库配置校验修正

## 为什么要做

`/health/ready` 现在会真实执行数据库就绪检查，这一步把一个原本就存在但此前未被稳定暴露的问题带出来了：

- API 与 worker 的启动入口当前没有可靠加载 monorepo 根目录 `.env`
- `libs/database` 在创建 Prisma / `pg` 连接时直接读取 `process.env.DATABASE_URL`
- 当 `DATABASE_URL` 没有被正确注入、为空值，或带有异常格式时，进程不会在启动阶段给出清晰配置错误，而是会在第一次真实数据库访问时，以底层 `pg` / SCRAM 报错的形式暴露出来

这会直接导致当前的开发体验出现误导：

- README 指导开发者复制 `.env` 后直接执行 `pnpm dev-api` / `pnpm dev-worker`
- 但运行时本身并未确保该 `.env` 会被加载到 Nest 进程
- 于是健康检查看起来像是“修坏了数据库”，实际上是 health 首次触发了一个本来就存在的运行时配置缺陷

因此需要用一个小 change，把“运行时环境加载”和“数据库连接配置校验”收敛到一个明确、可复用、可失败快速的基础行为里。

## 本次变更包含什么

本次变更聚焦以下内容：

- 为 `apps/api` 与 `apps/worker` 建立一致的本地运行时环境加载方式
- 明确 monorepo 根 `.env` 在本地开发中的发现与加载规则
- 为 `libs/database` 增加 `DATABASE_URL` 的最小规范化与显式校验
- 要求数据库配置错误在应用启动阶段即失败，并给出清晰错误信息
- 同步更新 README、必要注释和测试计划

## 本次变更不包含什么

本次变更不包含以下内容：

- 完整的全局配置中心或 `libs/config` 基建
- 所有环境变量的一次性全面建模
- secrets manager、远程配置中心或生产环境密钥托管方案
- 对 health 端点契约本身的再次改造

## 预期结果

完成后，项目应满足以下结果：

1. 在本地开发模式下，`apps/api` 与 `apps/worker` 能稳定加载仓库根 `.env`
2. `DATABASE_URL` 缺失、为空、带异常包裹字符或不可解析时，应用应在启动阶段快速失败
3. `/health/ready` 不再成为首次暴露数据库配置错误的入口
4. README 的本地启动说明与真实运行时行为一致

## 影响范围

本次变更主要影响：

- `apps/api` 与 `apps/worker` 的启动与配置加载方式
- `libs/database` 的连接字符串读取与校验逻辑
- 根 README 中的本地开发与环境变量说明
