# 变更提案：P0 短信共享层基建

## 为什么要做

`auth` 首期能力明确要求支持手机号验证码登录，而 `repository-structure` 真相源也已经要求任一业务能力发送短信时都必须通过 `libs/sms` 暴露的共享抽象接入。

但当前仓库中的 `libs/sms` 仍然是占位状态，还没有形成真正可复用的短信 port、provider 选择、调试替身和基础日志接线能力。

如果在这种状态下直接推进 `auth`，后续很容易再次出现上一次返工中暴露的问题：

- `auth` 模块内部私建 provider factory
- 供应商 SDK、模板参数和发送结果散落在业务模块里
- 本地开发与测试缺少稳定可重复的 mock / fake provider
- observability、rate limit、错误分类和审计字段无法统一
- 后续其他短信场景无法复用同一层基础设施

因此需要先建立一套首期可复用的短信共享层基建。

## 本次变更包含什么

本次变更聚焦于 `libs/sms` 共享运行时基础层，范围包括：

- 定义 `libs/sms` 的职责边界，明确其负责短信发送 port、provider adapter、provider 选择和调试替身
- 建立统一的短信发送请求与发送结果模型
- 约定本地开发与测试可用的 fake / debug provider 行为
- 明确与 `auth`、observability 的边界，保证业务模块依赖共享抽象而不是直接依赖供应商 SDK
- 约定首期错误分类、最小日志和脱敏边界
- 要求同步补齐 `libs/sms` README、必要注释和相关入口文档

## 本次变更不包含什么

本次变更不包含以下内容：

- `auth` 验证码完整业务流程
- 短信频控、验证码缓存与会话逻辑
- 短信发送异步化或 queue 集成
- 多供应商智能切换、自动降级与高级路由策略
- 营销短信、通知短信、国际短信等全部场景的一次性实现

## 预期结果

完成后，项目应具备以下结果：

1. 仓库中存在一套可复用的 `libs/sms` 共享运行时基础层
2. `auth` 等业务能力可以通过统一 port 发送短信，而不是在模块内自行接供应商
3. 本地开发与测试可以使用统一的 fake / debug provider
4. 发送结果、错误分类、日志和脱敏边界有统一约定
5. 后续如果需要把短信发送异步化，可以在此基础上再接 queue，而不必重写短信共享层

## 影响范围

本次变更主要影响：

- 新增的 `sms` capability
- `libs/sms` 的职责边界、README 与运行时接线方式
- `auth` 等依赖短信发送的后续业务 change
- 未来可能依赖短信发送的 queue / worker 任务
