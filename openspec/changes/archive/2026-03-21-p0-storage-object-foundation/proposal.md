# 变更提案：P0 对象存储共享层基建

## 为什么要做

当前仓库已经预留了 `libs/storage`，但它仍然停留在占位状态，还没有形成真正可复用的对象存储抽象与 provider 接线能力。

与此同时，现有真相源已经明确首期内容系统需要支持图片与封面媒体：

- `content` 首期帖子至少支持图片与封面
- `feed` 内容卡片需要返回图片或封面

如果在没有共享存储层的情况下直接推进 `content`、`feed` 或后续 Agent 资料相关能力，后续很容易出现：

- 业务模块内部各自接对象存储 SDK
- 上传结果、访问 URL、文件标识与元信息边界长期漂移
- 本地开发与测试没有统一 fake / local provider
- 媒体文件命名、目录前缀、公开访问与受控访问策略缺少统一约束
- 后续图片处理、审核、异步转码难以建立在稳定基线上

因此需要先通过一个独立的小 change，建立首期对象存储共享层基建。

## 本次变更包含什么

本次变更聚焦于 `libs/storage` 共享运行时基础层，范围包括：

- 定义 `libs/storage` 的职责边界，明确其负责对象存储 port、provider adapter、访问 URL 与最小文件元信息抽象
- 建立统一的上传请求、存储对象标识和读取结果模型
- 约定本地开发与测试使用的 fake / local provider 行为
- 明确业务模块与共享存储层的边界，保证 `content` 等能力依赖共享抽象而不是直接依赖供应商 SDK
- 约定首期最小公开访问 URL 或等价文件访问结果边界
- 要求同步补齐 `libs/storage` README、必要注释和相关入口文档

## 本次变更不包含什么

本次变更不包含以下内容：

- 完整媒体上传业务流程
- 图片压缩、裁剪、转码、水印等媒体处理流水线
- CDN、签名上传、断点续传与高级权限控制的完整建设
- 媒体审核、异步处理和队列接线
- `content`、`feed`、`agents` 等业务模块的一次性全面接入

## 预期结果

完成后，项目应具备以下结果：

1. 仓库中存在一套可复用的 `libs/storage` 共享运行时基础层
2. `content`、`feed` 等能力后续可以通过统一存储抽象接入对象存储
3. 本地开发与测试可以使用统一的 fake / local provider
4. 文件标识、访问 URL、对象命名和最小元信息边界有统一约定
5. 后续如果要接媒体处理、审核或异步化流程，可以建立在这套共享基线上继续扩展

## 影响范围

本次变更主要影响：

- 新增的 `storage` capability
- `libs/storage` 的职责边界、README 与运行时接线方式
- `content`、`feed` 以及后续依赖媒体文件的业务 change
- 未来可能依赖对象存储的 queue / worker 任务
