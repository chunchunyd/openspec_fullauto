# 变更提案：Auth Step 02 验证码发送与冷却控制

## 为什么要做

在 auth 主流程中，发送验证码是最早对外暴露的交互入口。

仓库当前已经有可复用的 `libs/sms` 和 `libs/cache`，适合直接承接验证码发送能力；但 `auth` 自身还没有：

- 手机号归一化和格式校验
- 登录验证码生成规则
- 冷却时间与发送频控
- 统一的发送接口 contract

如果跳过这一步直接推进登录校验或 token，会导致验证码生成、缓存和短信发送逻辑散落在后续 change 中，难以验证和复用。

因此需要先把“发码”这半段主流程单独落成一个小 change。

## 本次变更包含什么

本次变更聚焦验证码发送链路，范围包括：

- 设计 auth 场景下的手机号归一化与基础校验
- 建立验证码生成与短期缓存逻辑
- 复用 `libs/sms` 发送登录验证码
- 建立发送冷却时间和基础发送频控
- 定义发送验证码的 API contract，并在需要时执行 `openapi-export`

## 本次变更不包含什么

本次变更不包含以下内容：

- 验证码校验与用户识别
- token 签发与 refresh
- 协议确认 gating
- 设备会话管理
- 更复杂的异常请求识别与设备风险判断

## 预期结果

完成后，项目应具备以下结果：

1. 用户可以请求登录验证码
2. 系统会按受控规则生成、缓存并发送验证码
3. 冷却时间和基础发送频控不再依赖临时逻辑
4. 后续验证码校验 step 可以复用这一发送链路和缓存契约

## 影响范围

本次变更主要影响：

- `apps/api/src/modules/auth`
- `libs/cache` 在 auth 领域的使用方式
- `libs/sms` 在 auth 场景的接入方式
- `packages/api_contracts` 导出的 auth 相关 OpenAPI 产物
