# 设计说明：Auth Step 10 验证码校验与会话建立风控补齐

## 目标

这个 step 只解决一个问题：把 auth 风控从“仅发送前置限制”补齐到“发送、校验、建会话”三段闭环。

重点有三件事：

1. 在验证码校验阶段接入最小风险判定
2. 在会话建立阶段接入最小风险判定
3. 为命中风险分支补结构化日志和测试

## 边界

本次设计覆盖：

- `OtpService.verifyLoginOtp()`
- `SessionService.createSession()`
- `RiskService` 已有信号采集与判定能力在 verify / session 阶段的复用
- 风险限制和风险拒绝场景的最小测试与日志

本次设计不覆盖：

- 新的风险模型或评分系统
- 额外数据库表或专门的风控事件流水
- 多端统一设备指纹识别
- `apps/api` 其他模块的安全治理

## 设计思路

### 1. verify 阶段复用现有风险信号采集能力

当前 `sendLoginOtp()` 已经能基于手机号、IP、UA、设备信息进行最小风险判断。

本 step 建议：

- 在 `verifyLoginOtp()` 增加可选上下文输入，至少允许带入 `clientIp`、`userAgent`、`deviceId`、`deviceOs`
- 复用 `RiskService.collectSignals()` 和 `RiskService.evaluateRisk()`
- 在验证码比对成功但建立用户或返回已验证结果前执行风险判定

这样可以避免“验证码已经正确，就直接无条件放行”的情况。

### 2. createSession 阶段增加最后一道风控门

即使 verify 阶段已经完成风险判断，会话建立仍是最值得再加一道最小门的地方，因为这是最终发放登录态的时刻。

本 step 建议：

- `createSession()` 接收风险相关上下文
- 在真正创建 session 并签发 token 前执行最小风险判定
- 若命中受控拒绝，则不创建会话、不签发 token
- 若命中受控限制，则按设计返回明确受限结果或执行更严格约束

是否在 verify 和 session 两段都判定，可以共享同一套信号采集，但不能只保留其中一个阶段。

### 3. 风险分支必须保留最小结构化日志

这个 step 不追求完整审计平台，但要确保后续能排查：

- 是 verify 阶段被拒绝还是 session 阶段被拒绝
- 命中的最小风险原因是什么
- 对应的手机号后四位、设备上下文和请求时间是否可见

日志仍需遵守现有要求：

- 不泄露完整手机号
- 不泄露验证码
- 不泄露 access / refresh token

## 关键失败场景

- 如果只在 verify 阶段补风控，不在 session 阶段兜底，仍可能出现“校验通过后签发会话前缺少最后一道限制”
- 如果只在 session 阶段补风控，不在 verify 阶段留痕，会让验证码校验阶段的异常行为继续不可见
- 如果风险分支只返回失败结果，不记结构化上下文，后续排查仍然困难
- 如果测试仍只验证 send 阶段，剩余高风险缺口会继续被忽略

## 与现有 specs 的关系

这个 step 是对已存在 auth 风控承诺的补齐，不是新增一个全新 capability。

它直接承接：

- `auth-step-07-risk-hardening` 中“发送 / 校验 / 会话流程接入最小风险信号”的目标
- 当前 `auth` spec 中“记录并判定基础登录风险信号”的承诺

完成后，`auth` 才能更一致地声称自己具备了最小可用的登录风险闭环。
