# 变更提案：Auth Step 03 验证码校验与账号识别

## 为什么要做

发码链路完成后，下一步需要把“收到验证码之后如何校验并识别账号主体”独立收敛出来。

如果直接把验证码校验、用户创建、账号状态判断、token 签发和协议 gating 一次性揉在同一个 change 中，会出现：

- 失败场景混杂，难以测试
- 用户识别与会话签发耦合过深
- 账号状态限制无法先独立验证

因此需要先把“验证码校验 + 账号识别 / 首次创建”这一段 auth 领域核心逻辑单独落下。

## 本次变更包含什么

本次变更聚焦验证码校验与账号识别，范围包括：

- 校验验证码是否正确、是否过期、是否已失效
- 建立验证码校验失败次数控制
- 为首次手机号创建最小可用账号，或识别已有账号
- 在进入后续会话签发前判断账号状态是否允许继续
- 产出可供下一步会话签发消费的已验证 auth 结果或等价领域服务

## 本次变更不包含什么

本次变更不包含以下内容：

- access token / refresh token 的正式签发
- refresh 接口
- 协议确认 gating
- 设备会话列表与退出登录接口
- 更复杂的设备或请求来源风险识别

## 预期结果

完成后，项目应具备以下结果：

1. 系统能够校验用户提交的登录验证码
2. 首次手机号可被识别并初始化为最小可用账号
3. 已存在账号可被识别并执行账号状态判断
4. 后续 step 可以围绕稳定的“已验证身份结果”继续签发会话

## 影响范围

本次变更主要影响：

- `apps/api/src/modules/auth`
- auth 领域服务中的验证码校验与用户识别逻辑
- `prisma/` 中 auth 主数据的实际使用方式
- 后续 `auth-step-04-session-token-and-refresh`
