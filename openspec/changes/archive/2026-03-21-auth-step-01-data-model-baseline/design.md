# 设计说明：Auth Step 01 认证数据模型基线

## 目标

这个 step 只做一件事：给 auth 后续能力建立稳定的数据落位。

它不试图一次性完成完整登录流程，而是把后续所有 auth step 都会依赖的主数据先定下来。

## 边界

本次设计覆盖：

- 用户账号状态
- 协议 / 告知确认记录
- 设备会话与 refresh token 持久化锚点

本次设计不覆盖：

- 验证码发送与缓存
- token 签发算法
- controller / API contract 细节
- 登录风控规则细节

## 建议的数据模型方向

### 1. 用户账号状态

`User` 至少需要能够表达：

- 手机号唯一标识
- 账号当前状态，例如 `active` / `banned` / `deactivated`
- 与 auth 主流程直接相关的最小审计字段

这里的重点不是把所有个人资料一次补齐，而是让“能否登录”有稳定判断基础。

### 2. 协议确认记录

协议确认建议使用独立记录，而不是把多个布尔字段硬塞进 `User`：

- 便于区分不同确认项
- 便于记录确认时间、版本和来源
- 后续如果协议版本更新，不必重做用户主表结构

首期至少需要覆盖：

- 用户协议
- 隐私政策
- AI 内容告知

### 3. 设备会话与 refresh token

设备会话建议独立为 `AuthSession` 或等价模型，至少应包含：

- 会话 ID
- 用户 ID
- 设备标识或设备摘要
- refresh token 摘要值，而不是明文
- 最近活跃时间
- 当前状态

这样可以同时支撑：

- step-04 的 token 签发与刷新
- step-06 的设备列表、会话移除和退出登录

## 设计原则

### 1. 只建立锚点，不提前塞满业务细节

这个 step 的目标是为后续能力提供稳定持久化锚点，而不是把所有 auth 逻辑都塞进 schema。

### 2. token 明文不落库

refresh token 如果需要持久化，建议只保存摘要值或等价安全表示，用于校验、轮换和撤销。

### 3. 与 `prisma/` 和 `libs/database` 保持清晰边界

- schema 真相源只在 `prisma/`
- 运行时数据库访问只通过 `libs/database`
- `apps/api` 的 auth 模块不应私建第二套 schema 真相源

## 失败场景

- 如果 schema 只补用户状态，不补会话锚点，后续 step-04 / step-06 还会再次返工
- 如果协议确认直接散落在多个布尔字段中，后续版本化和审计会变得很难收敛
- 如果 refresh token 明文落库，会引入不必要的安全风险
