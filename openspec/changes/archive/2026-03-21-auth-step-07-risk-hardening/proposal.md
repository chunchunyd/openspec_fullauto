# 变更提案：Auth Step 07 登录风险信号加固

## 为什么要做

在验证码发送、验证码校验和会话机制都具备之后，auth 还需要补上最低限度的风险信号与异常限制。

虽然前面 step 已经覆盖了基础发送频控和失败次数控制，但 auth spec 还要求系统能对异常设备或异常请求提供基础识别与拦截能力。

如果这部分一直留在口头上：

- auth 只能处理“明显错误”，却无法对异常来源做最小区分
- 日志和风控信号无法形成可排查的 auth 留痕
- 后续后台治理、设备会话管理与异常登录排查难以衔接

因此需要最后用一个小 step，把登录风险信号与受控限制补齐。

## 本次变更包含什么

本次变更聚焦基础登录风险信号，范围包括：

- 识别并记录最小请求来源或设备信号
- 将这些信号与验证码发送 / 校验失败形成联动限制
- 为异常登录链路补必要日志和错误上下文
- 明确哪些异常结果属于受控拒绝，哪些只是留痕

## 本次变更不包含什么

本次变更不包含以下内容：

- 复杂设备指纹系统
- 多维度风控平台
- 后台审核工作台
- 第三方风险引擎接入

## 预期结果

完成后，项目应具备以下结果：

1. auth 能记录并利用最小请求来源或设备信号
2. 明显异常来源会受到基础限制或受控拒绝
3. 登录异常不再只剩“验证码错了”这一种粗糙结果
4. 后续治理与审计可以基于这些风险信号继续扩展

## 影响范围

本次变更主要影响：

- `apps/api/src/modules/auth`
- `libs/observability` 在 auth 场景中的接线
- 验证码发送 / 校验与会话流程中的风险判断分支
