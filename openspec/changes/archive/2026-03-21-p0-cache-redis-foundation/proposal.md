# 变更提案：P0 Redis 基建与缓存分层基线

## 为什么要做

在继续推进 `auth` 返工以及后续更多业务能力前，项目需要先建立一套明确、可复用的 Redis 基建和缓存分层规则。

当前仓库虽然已经预留了 `libs/cache` 目录，但共享缓存层尚未形成稳定可复用的运行时基础设施，也没有把“Redis 基础设施”“业务缓存键定义”“跨运行时共享缓存契约”三层职责真正分开。

如果在这种状态下直接推进 `auth`、通知、任务调度或其他依赖 Redis 的能力，后续很容易出现以下问题：

- 业务模块各自创建原生 Redis client 或重复包装一套缓存接线
- 验证码、冷却时间、频控计数等短期状态被随意落在数据库或本地实现中
- 缓存 key 命名、TTL、值结构和环境隔离规则不统一
- 业务模块自己的缓存语义被错误收口到 `libs/cache`，最终形成难以维护的“全局 key 大杂烩”
- 多运行时真正需要共享同一套缓存契约时，没有明确的提取时机和落位规则

因此需要先通过一个独立的小 change，建立 Redis 运行时基建与缓存分层基线，为后续业务 change 提供稳定依赖。

## 本次变更包含什么

本次变更聚焦于 Redis 基建与缓存分层规范，范围包括：

- 定义 `libs/cache` 的职责边界，明确它负责 Redis 连接、共享运行时封装、键名辅助和通用缓存原语
- 定义业务模块本地缓存目录的职责边界，明确业务 key、TTL 和值结构优先放在各自模块目录下维护
- 定义未来跨运行时共享缓存契约的提取规则，明确何时应升级为 `packages/cache_contracts`
- 为 Redis key 设计建立统一命名规范，覆盖环境隔离、领域命名、版本化和敏感标识处理
- 约定首期最小共享缓存原语，以支撑验证码、冷却时间、频控计数等典型短期状态场景
- 要求同步补齐 `libs/cache` README、必要代码注释，以及与 Redis 启动/配置相关的入口文档

## 本次变更不包含什么

本次变更不包含以下内容：

- 具体的 `auth` 验证码登录实现
- 具体业务模块的 Redis key 落地实现
- 消息队列、任务投递或 worker 消费逻辑
- 分布式锁、发布订阅、二级缓存、缓存失效广播等高级 Redis 模式
- 所有业务缓存策略的一次性统一建设

## 预期结果

完成后，项目应具备以下结果：

1. 仓库中存在一套可复用的 `libs/cache` 共享运行时基础层
2. Redis key 设计有统一规范，但业务 key 的 ownership 保留在各业务模块内
3. 后续如 `auth` 需要验证码、冷却时间、频控计数等能力时，可以直接建立模块内缓存目录并复用 `libs/cache`
4. 当未来 `api`、`worker` 或多个运行时需要共享同一套缓存 key 与值结构时，团队知道何时提取到 `packages/cache_contracts`
5. Redis 相关的 README、注释和入口文档要求被一起固化，而不是等实现结束后再补

## 影响范围

本次变更主要影响：

- `repository-structure` capability
- 新增的 `cache` capability
- `libs/cache` 的目录设计、README 与共享运行时边界
- 后续所有依赖 Redis 的业务 change，尤其是 `auth`
