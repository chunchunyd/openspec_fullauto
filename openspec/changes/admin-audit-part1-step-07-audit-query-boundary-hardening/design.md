## Context

当前后台审计中心已经具备最小读取入口，但它的“受控查询边界”仍然存在实现层偏差：

- `startTime` 与 `endTime` 分别展开各自的 `createdAt` 条件，导致二者同时存在时前者被后者覆盖
- `sortBy` / `sortOrder` 在文档上有限制，但运行时只按普通字符串处理，随后直接传入动态 `orderBy`

这类问题不会改变 step-02 的总体目标，却会让查询结果或错误语义在真实使用时变得不稳定。

## Goals / Non-Goals

**Goals**

- 让审计时间范围条件可以安全组合
- 让排序参数在 DTO 与仓储层都保持白名单语义
- 补足能证明这两类问题被真正修复的回归测试

**Non-Goals**

- 不新增全文搜索、复杂筛选器或 metadata 公开字段
- 不改动审计记录 owner、写入流程或 analytics 派生边界

## Decisions

### 1. 时间范围统一收敛到单个 `createdAt` 条件对象

本 step 会把 `startTime` / `endTime` 的组合统一写入同一个 `createdAt` 查询对象，使：

- 只传开始时间时，生效 `gte`
- 只传结束时间时，生效 `lte`
- 同时传入时，两个条件一起生效

### 2. 排序参数在 DTO 和仓储层双层受控

本 step 不会继续信任任意字符串排序字段，而是采用：

- DTO 层白名单校验
- 仓储层字段映射或等价受控转换

这样即使未来调用方绕过文档，也不会让非法字段直接落到 Prisma 动态 `orderBy`。

### 3. 保持现有查询能力边界不外扩

本 step 只收口 step-02 已经承诺的“多条件筛选、分页和排序”主线，不把这次修复顺手扩大成“后台审计中心 v2”。

## Risks / Trade-offs

- [排序字段白名单会收紧调用方自由度] -> 这是受控查询边界本来就应承受的约束，优于让非法入参在仓储层炸成 500
- [测试需要下沉到更接近仓储/控制器组合] -> 这是为了证明“不是只把参数传下去，而是真的能正确组合条件”

## Migration Plan

本 step 不涉及 schema 迁移。实施重点是先修正 DTO / repository，再通过测试证明时间区间与排序白名单都在运行时生效。

## Open Questions

- 如果实现时发现后台还需要对 `action` 的模糊匹配做额外约束，应只记录为后续问题，不在本 step 继续扩展范围。
