## Context

step-04 已经把后台用户治理动作接入了最小权限校验与审计记录，但当前状态流转仍然把“封禁”当成对 `User.status` 的单列覆盖。

这会导致一个关键问题：

- `ACTIVE/DEACTIVATED -> BANNED`
- `BANNED -> ACTIVE`

当原始状态是 `DEACTIVATED` 时，解封会错误“复活”账号。这个问题不是简单的 if 分支疏漏，而是当前状态机缺少“封禁前状态恢复锚点”。

## Goals / Non-Goals

**Goals**

- 让 ban / unban 形成受控状态机，而不是覆盖式状态写入
- 为解封提供恢复封禁前状态的最小锚点
- 在恢复锚点缺失或状态非法时返回受控失败

**Non-Goals**

- 不引入申诉、审批、自动风控等新流程
- 不在本 step 同时处理用户读取门禁和审计查询排序问题
- 不把 audit log 变成用户主状态的权威 owner

## Decisions

### 1. 引入“封禁前状态恢复锚点”

本 step 会为用户主数据补充一个最小恢复锚点，例如 `statusBeforeBan` 或等价字段，使系统在执行 ban 时记录：

- 用户封禁前的原始主状态
- 当前封禁动作对应的最小恢复依据

解封时优先恢复该锚点，而不是默认写回 `ACTIVE`。

### 2. ban / unban 继续保持受控幂等

本 step 不移除 step-04 已有的幂等目标，但会把幂等建立在受控状态机之上：

- 已封禁用户再次封禁：返回幂等成功，不重复破坏恢复锚点
- 已恢复到目标状态的用户再次解封：返回幂等成功或受控无操作结果

### 3. 恢复锚点缺失时返回受控失败

对于历史遗留或异常数据，如果用户处于 `BANNED` 但缺少恢复锚点，本 step 不会继续默认写回 `ACTIVE`。

系统应返回受控失败结果，要求人工排查或通过后续专门治理路径处理，而不是在运行时静默制造新的状态污染。

## Risks / Trade-offs

- [需要引入最小 schema 变更] -> 这是换取正确状态恢复语义的必要成本，优于继续让后台治理覆盖其他业务 owner
- [需要考虑历史 `BANNED` 数据] -> 比起默认回写 `ACTIVE`，显式失败更安全，也更利于后续人工处理

## Migration Plan

本 step 预计涉及最小 schema 变更。实施时应：

1. 先补恢复锚点字段与等价持久化能力。
2. 再改 ban / unban 状态机。
3. 最后补历史缺失锚点时的受控失败测试。

## Open Questions

- 如果实现时发现现有仓库更适合通过独立治理快照表保存恢复锚点，可在不扩大 change 主目标的前提下替换字段落点，但仍必须保证“解封不默认写回 `ACTIVE`”。
